# 배운 점 및 회고 (Learnings)

React Native 하이브리드 앱 개발을 통해 기술적인 성장뿐만 아니라 아키텍처 설계에 대해 깊이 고민할 수 있었습니다.

---

## 1. 하이브리드 아키텍처의 가치

**"최선의 도구로 최선의 문제를 해결하라"**

처음에는 "제대로 된 앱이라면 순수 네이티브로 개발해야 한다"는 생각이 있었습니다.
하지만 이 프로젝트를 통해 **하이브리드 접근이 때로는 더 현명한 선택**임을 깨달았습니다.

- **핵심 문제 식별**: 인앱 브라우저 OAuth 차단, 홈 위젯, 푸시 알림 - 이 세 가지만 네이티브로 해결하면 됨
- **코드 재사용**: 검증된 웹앱을 버리고 처음부터 다시 개발하는 것은 비효율적
- **유지보수 용이**: 한 번 수정하면 웹/앱 모두 반영

**결론**: 기술 선택은 "어떤 기술이 좋은가"가 아니라 "어떤 문제를 해결하는가"에 따라 결정해야 합니다.

---

## 2. 두 세계의 경계에서 배운 것

**Native와 Web 사이의 브릿지 설계는 생각보다 복잡했습니다.**

- **상태 동기화의 어려움**: 인증 상태, 테마, 언어가 양쪽에서 변경될 수 있음
- **타이밍 이슈**: `injectedJavaScript` vs `injectedJavaScriptBeforeContentLoaded`의 차이가 FOUC를 결정
- **데이터 포맷 불일치**: JavaScript 객체 ↔ Kotlin data class 변환 필요

이 경험을 통해 **인터페이스 설계의 중요성**을 배웠습니다.
명확한 메시지 타입과 payload 구조를 먼저 정의한 덕분에 복잡한 양방향 통신을 체계적으로 관리할 수 있었습니다.

---

## 3. Android 네이티브 영역 탐험

**Kotlin과 Android SDK 경험은 프론트엔드 개발자의 시야를 넓혀주었습니다.**

- **AppWidgetProvider**: RemoteViews, SharedPreferences, PendingIntent 등 Android 위젯 시스템 학습
- **AlarmManager**: 정확한 시간에 알림을 트리거하는 시스템 알람 관리
- **BroadcastReceiver**: BOOT_COMPLETED, TIME_SET 등 시스템 이벤트 처리

웹 개발만 하던 입장에서 "OS 레벨에서 이런 일이 일어나는구나"를 직접 체험할 수 있었습니다.
특히 **위젯의 제약 사항**(동적 UI 업데이트 불가, RemoteViews만 사용 가능)을 이해하는 것이 중요했습니다.

---

## 4. 보안에 대한 고민

**모바일 앱에서의 토큰 관리는 웹과 다릅니다.**

- **EncryptedStorage**: 일반 AsyncStorage 대신 암호화된 저장소 사용
- **토큰 노출 최소화**: `injectedJavaScript`로 토큰을 주입할 때, 로그에 노출되지 않도록 주의
- **ID Token vs Access Token**: Google에서 받은 ID Token으로 자체 Access Token을 발급하는 흐름

보안은 "나중에 하자"가 아니라 **설계 단계부터 고려**해야 한다는 것을 배웠습니다.

---

## 5. 사용자 경험의 디테일

**네이티브 앱에서는 작은 디테일이 큰 차이를 만듭니다.**

- **백버튼 처리**: 웹에서는 신경 쓰지 않던 하드웨어 백버튼
- **앱 상태 관리**: 백그라운드 ↔ 포그라운드 전환 시 상태 복원
- **스플래시 화면**: 초기 로딩 동안 브랜드 스플래시로 체감 대기 시간 감소

"웹에서는 당연히 동작하던 것"이 모바일에서는 직접 구현해야 하는 경우가 많았습니다.
